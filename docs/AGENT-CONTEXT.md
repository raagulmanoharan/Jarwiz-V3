# What the Agent Sees

A narrative + real JSON walkthrough of exactly what goes to Claude when you summon an agent. Read this before touching anything in `apps/server/src/agents/`.

The example throughout is: you drop a PDF onto the canvas, select it, and click **"What should I worry about here?"** — which summons the Summarizer.

---

## The trigger

The user presses Enter in the prompt bar with a `pdf-card` selected. The web client calls:

```
POST /api/agents/summarizer/run
```

with this body:

```json
{
  "source": {
    "cardId": "shape:2g4kHq9vMz",
    "kind": "pdf",
    "x": 320,
    "y": 180,
    "w": 480,
    "h": 640,
    "url": "https://assets.jarwiz.app/uploads/pdf/abc123.pdf",
    "title": "McKinsey Global Institute — The State of AI 2025.pdf"
  },
  "selection": [],
  "placement": { "x": 860, "y": 180 },
  "brief": "What should I worry about here?"
}
```

**`source`** is the seed card — the card the agent was summoned from. **`selection`** would contain any additional cards the user had selected simultaneously (here, none). **`placement`** is the nearest free space to the right of the PDF card, computed by the client. **`brief`** is the user's typed instruction.

The server routes this to `apps/server/src/agents/summarizer.ts`.

---

## What the Summarizer sends to Claude

The API call to Anthropic is a standard `messages.create` with two payload sections: a frozen system prompt and a volatile user turn.

### System prompt (frozen for prompt caching)

The system prompt never changes between runs. It's a static string that ends with `cache_control: { type: 'ephemeral' }`, so Anthropic's infrastructure caches it after the first call and reuses the KV cache on subsequent runs — shaving ~500ms and significant cost.

```json
{
  "role": "user",
  "content": [
    {
      "type": "text",
      "text": "You are the Jarwiz Summarizer...\n[~400 token static instruction block]",
      "cache_control": { "type": "ephemeral" }
    }
  ]
}
```

The system prompt instructs the agent to:
1. Fetch the source URL with `web_fetch` (except YouTube — oEmbed only)
2. Create a doc card with `begin_card` at the placement hint
3. Stream a 120–220 word summary in markdown (no headers, no lists — flowing prose)
4. Close the card with `finish_card`
5. Connect the source to the artifact with `connect_cards`
6. Be honest about gated or inaccessible content

### User turn (volatile — built fresh each run)

The user turn is generated by `buildUserTurn(request)` in `summarizer.ts`. For our example:

```
Summarize the source card below onto the board.

Source card:
  cardId: shape:2g4kHq9vMz
  kind: pdf
  position: x=320, y=180 (w=480, h=640)
  url: https://assets.jarwiz.app/uploads/pdf/abc123.pdf
  title: McKinsey Global Institute — The State of AI 2025.pdf
  text: """"""

Placement hint (free space on the board): put your card's top-left at x=860, y=180.
Remember to connect_cards from "shape:2g4kHq9vMz" to your new card when you are done.

Focus: What should I worry about here?
```

The `text` field is empty because PDFs don't embed their body text in the card props — the agent is expected to fetch the URL. For a `doc-card` or `note-card`, `text` would contain the full body.

The `Focus:` line is appended by `briefSuffix()` when a brief is present. Without a brief, it's omitted entirely, and the agent applies its default summarization behavior.

## The tools Claude can call

The runtime exposes two categories of tools: **canvas tools** (always present) and **server tools** (agent-specific).

### Canvas tools (all agents)

These are the tools Claude uses to place artifacts on the board. The runtime intercepts each `tool_use` block and translates it into `AgentEvent`s that stream to the client.

**`begin_card`** — Opens a new streaming doc or note card.
```json
{
  "name": "begin_card",
  "input": {
    "kind": "doc",
    "x": 860,
    "y": 180,
    "title": "AI 2025: What to Watch"
  }
}
```
The runtime responds with `{ cardId: "card_1", next: "Write the card body now..." }` and emits:
```json
{ "type": "card.create", "cardId": "card_1", "kind": "doc", "x": 860, "y": 180, "title": "AI 2025: What to Watch" }
```

**`finish_card`** — Closes an open card and stops the streaming caret.
```json
{ "name": "finish_card", "input": { "cardId": "card_1" } }
```
Emits: `{ "type": "card.done", "cardId": "card_1" }`

**`create_link_card`** — Places a URL preview card (Researcher only).
```json
{
  "name": "create_link_card",
  "input": {
    "url": "https://example.com/article",
    "title": "The article title",
    "description": "One-line description",
    "x": 860,
    "y": 360
  }
}
```

**`create_note`** — Places a sticky note (Brainstormer primarily).
```json
{ "name": "create_note", "input": { "text": "What if we inverted the assumption?", "x": 860, "y": 180 } }
```

**`create_table`** — Places a grid card.
```json
{
  "name": "create_table",
  "input": {
    "columns": ["Risk", "Likelihood", "Mitigation"],
    "rows": [["Model concentration", "High", "Multi-vendor contracts"]],
    "x": 860,
    "y": 180,
    "title": "Risk Register"
  }
}
```

**`connect_cards`** — Draws a provenance arrow.
```json
{
  "name": "connect_cards",
  "input": {
    "fromCardId": "shape:2g4kHq9vMz",
    "toCardId": "card_1",
    "label": "source"
  }
}
```
Emits: `{ "type": "edge.create", "fromCardId": "shape:2g4kHq9vMz", "toCardId": "card_1", "label": "source" }`

The client maps `"card_1"` to a real tldraw `TLShapeId` via the run-scoped `idMap`.

### Server tools (Summarizer, Researcher, Writer)

**`web_fetch_20260209`** — Fetches a URL and returns its text content. The Summarizer uses this to read the PDF (up to 3 calls per run). The Researcher uses it to vet search results.

**`web_search_20260209`** — Issues a web search and returns results. Researcher only, up to 4 calls per run.

The Brainstormer has no server tools — it's purely generative.

## How streaming text reaches the canvas

Between `begin_card` and `finish_card`, Claude emits text as normal streaming deltas. The runtime intercepts these at the `stream.on('text')` handler and relays them as `card.delta` events:

```json
{ "type": "card.delta", "cardId": "card_1", "textDelta": "The report identifies " }
{ "type": "card.delta", "cardId": "card_1", "textDelta": "three structural risks " }
{ "type": "card.delta", "cardId": "card_1", "textDelta": "in enterprise AI adoption..." }
```

The client appends each delta to the doc card's body in real time. This is what makes the writing appear live — the streaming caret is driven by the `streaming.ts` external store, which tracks whether a card id is currently open.

The runtime serializes text and tool events to prevent interleaving: if a `tool_use` block arrives while text is still buffered, the flush happens before the tool executes.

## The complete event sequence

For our PDF → Summarizer example, the full SSE stream from the server looks like this:

```
data: {"type":"status","message":"Reading the document…"}

data: {"type":"cursor","x":560,"y":500}

data: {"type":"card.create","cardId":"card_1","kind":"doc","x":860,"y":180,"title":"AI 2025: What to Watch"}

data: {"type":"card.delta","cardId":"card_1","textDelta":"The McKinsey report identifies "}
data: {"type":"card.delta","cardId":"card_1","textDelta":"three structural risks "}
data: {"type":"card.delta","cardId":"card_1","textDelta":"as AI scales across enterprise: "}
...
[~60 more delta events]
...

data: {"type":"card.done","cardId":"card_1"}

data: {"type":"cursor","x":860,"y":180}

data: {"type":"edge.create","fromCardId":"shape:2g4kHq9vMz","toCardId":"card_1","label":"source"}

data: {"type":"done"}
```

Each event is a Server-Sent Event on the path `/api/agents/summarizer/run`. The connection is kept open until `done` or `error` arrives.

## What changes per agent

All four agents receive the same canvas tools and the same user-turn serialization structure. The differences are:

| Agent | Server tools | Card types placed | Connection label | Approximate word count |
|---|---|---|---|---|
| Researcher | `web_search` × 4, `web_fetch` × 3 | `link-card` × 3–5 | "source" | N/A (metadata only) |
| Summarizer | `web_fetch` × 3 | `doc-card` × 1 | "source" | 120–220 words |
| Brainstormer | none | `note-card` × 5–7 | "idea" | ~20 words each |
| Writer | `web_fetch` × 5 | `doc-card` × 1 | "drawn from" | 250–500 words |

The Writer also connects from *every* input card (not just the seed) with label "drawn from".

## What the agent does NOT see

Understanding what's *excluded* is as important as what's included:

- **The whole board.** Agents only see the `source` and `selection` cards — not every shape on the canvas. If you want the agent to consider a card, select it before summoning.
- **Chat history.** Each agent run is stateless. There's no conversation thread; the user turn is built fresh each time from the current card state.
- **Other agents' system prompts.** Each agent's frozen system prompt is isolated.
- **The user's identity.** The runtime sends no user metadata to Claude.
- **tldraw internals.** The `RunCard` serialization only includes `{ cardId, kind, x, y, w, h, url?, title?, text? }`. Styling, bindings, and shape metadata are stripped.

## Runtime limits

- **Max tokens:** 16,000 per run
- **Max iterations:** 12 tool-use rounds
- **Model:** `claude-opus-4-8` with `thinking: { type: "adaptive" }` for all agents
- **Metadata cleanup:** `claude-haiku-4-5` for link card title/description normalization only

If the iteration cap is hit, the runtime emits an `error` event and the run ends. The partial artifacts already on the canvas stay — the user can continue from there manually.

See the full procedure in `CLAUDE.md`. The TL;DR: implement `buildUserTurn(request)`, write a static `systemPrompt`, export an `AgentDefinition`, register in `agentRun.ts`, add a mock branch in `mock.ts`. The runtime owns everything else.
