# Jarwiz — Pre-launch Stress Test & Verification Report

**Date:** 2026-07-07 · **Build:** `claude/app-testing-stress-report-ct2my1`
**Backend mode:** `sidecar` — real Claude via the Claude Code CLI, `live=true`
(no `ANTHROPIC_API_KEY`; genuine generation, not the scripted mock).
**Harness:** [`scripts/eval-stress-100.mjs`](../scripts/eval-stress-100.mjs) —
100 use cases across every backend endpoint, run through a bounded concurrency

---

## TL;DR

> **100 / 100 use cases pass** against the real (sidecar) runtime in a single
> canonical run (293s wall, concurrency 6). Every endpoint the web app calls
> is exercised and returns well-formed, genuinely-generated output. Typecheck
> and build are green across all three workspaces.

Three things stand between "all tests green" and "clean open-source launch" —
none is a functional break, all are worth fixing in the next few weeks:

1. **A deep-research Thinking Machine (SWOT) took ~154s** — right at the
   sidecar's 180s ceiling. A perf/UX risk on slow networks, not a failure.
2. **The canonical `scripts/eval-server.mjs` is stale** — it tests
   `POST /api/comment`, an endpoint that no longer exists (now 404). A fresh
   clone would see that suite fail, and `docs/FEATURES.md`'s "server 10/10"
   claim is out of date.
3. **Multiplayer is OFF by default** (`JARWIZ_ENABLE_SYNC` unset), yet
   `docs/FEATURES.md` lists it as a verified collaboration feature. Launch
   framing should call it experimental/gated.

## What was tested, and how

The server was booted in real `sidecar` mode and confirmed live via
`GET /api/capabilities → {"live":true,"mode":"sidecar"}`. Every case makes a
real HTTP call to `http://localhost:3001` and asserts the *shape* of what comes
back — for streaming endpoints, that the SSE stream produced real content
events and terminated cleanly; for JSON endpoints, that the payload matches the
contract; for validation cases, that bad input is rejected with the right
status.

"Real sidecar" matters: these are not mock responses. Every doc, table, list,
diagram, sticky, comparison, SWOT board and research dossier below was
generated by Claude at request time.

## Findings & recommendations

### 🟡 1 — Deep-web Thinking Machines flirt with the timeout ceiling
`sidecar.ts` caps web-enabled runs at **180s** (`WEB_TIMEOUT_MS`). The SWOT
board came back at **154s** — a 26s margin. On a slower network, or with the
5-card fan-out running each card serially, this can tip into a timeout the user
experiences as a dead spinner.

- **Recommend:** under a real `ANTHROPIC_API_KEY` (the production path) this is
  much faster — the CLI sidecar is a dev convenience. Still, before launch:
  (a) ensure the canvas shows visible progress for board machines (the `plan`
  event arrives early — render the empty card frames immediately, which it
  appears to do), and (b) consider raising the web ceiling or parallelizing the
  per-card generation in `machineBoard.ts`.

### 🔴 2 — `scripts/eval-server.mjs` is stale (tests a removed endpoint)
The canonical backend suite still asserts `POST /api/comment`, which now
returns **404** — it was replaced by `/api/notice` (proactive comments) and
`/api/annotate` (stickies). A fresh `git clone` running the advertised
`node scripts/eval-server.mjs` would fail. `docs/FEATURES.md` compounds this:
it claims **"server 10/10"** and lists *"Verified: `POST /api/comment`"*.
Stale references also live in `docs/DECISIONS.md` and `docs/RESPONSIVENESS.md`.

- **Recommend:** update `eval-server.mjs` to test `/api/notice` + `/api/annotate`
  instead of `/api/comment`, and refresh the `docs/FEATURES.md` verification
  claims. This is the single most important pre-open-source fix — the test
  suite is the first thing contributors run.

### 🟡 3 — Multiplayer is gated off, but documented as verified
The server boots with `multiplayer sync is parked pending security hardening
(set JARWIZ_ENABLE_SYNC=1 to enable)`. `docs/FEATURES.md` lists **Multiplayer
sync** under Collaboration with *"Verified: Multiplayer sync (2 clients)"*, and
`eval-server.mjs`'s WebSocket-upgrade check would fail in the default config.

- **Recommend:** reframe multiplayer as **experimental / opt-in** in
  `README.md` and `docs/FEATURES.md`, matching `docs/AUDIT.md P0.4`. Don't ship
  a launch claim that's off by default.

### 🟢 4 — Not covered here: canvas UI interactions
This report verifies the **backend contracts** behind every feature. It does
**not** click-test the tldraw canvas itself — drag/drop ingestion, live
streaming carets, Tab-to-autopilot keypresses, the ⌘K palette, provenance
edges. Those need a real browser (the sandbox's dev-HMR websocket is blocked;
see `CLAUDE.md`).

- **Recommend:** run `scripts/eval-ui.mjs` (Playwright, 12 checks) against a
  `vite preview` build in a normal environment, and do one manual pass of the
  drop→suggest→summon→autopilot loop before launch.

---

## Verdict

**Functionally, Jarwiz is launch-ready.** Every user-facing capability —
ingestion, the four agents, Autopilot, the Ask router across all seven card
shapes, the seven Thinking Machines, and the whole board-intelligence layer —
works end-to-end against real Claude, with sane input validation and working
SSRF guards. Typecheck and build are clean.

The three open items above are **documentation/framing accuracy and one perf
edge**, not broken features. Fixing the stale `eval-server.mjs` (Finding 2) and
the multiplayer framing (Finding 3) is the highest-leverage pre-launch work,
because both shape a new contributor's very first impression of the project.
